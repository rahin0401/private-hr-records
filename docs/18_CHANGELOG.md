# 1. Purpose

This document records all notable changes made to the Privacy-Preserving Synthetic HR Records Generator throughout its lifecycle.

The changelog provides a chronological history of releases, improvements, fixes, architectural changes, and new capabilities.

It is intended for developers, maintainers, testers, and stakeholders who need to understand how the platform has evolved over time.

---

# 2. Changelog Principles

The changelog shall:

* Record only notable changes.
* Follow chronological order.
* Group changes by release.
* Avoid implementation-level details better suited for commit history.
* Reference Architecture Decision Records (ADRs) where appropriate.

---

# 3. Change Categories

Each release should classify changes using the following sections:

### Added

New features or capabilities.

### Changed

Modifications to existing behavior.

### Deprecated

Features planned for removal in future releases.

### Removed

Features no longer supported.

### Fixed

Bug fixes and corrections.

### Security

Security improvements and vulnerability fixes.

### Performance

Performance enhancements and optimizations.

### Documentation

Updates to architecture or engineering documentation.

---

# 4. Versioning Policy

The project follows Semantic Versioning.

```text id="x5b4z2"
MAJOR.MINOR.PATCH
```

Examples:

* **MAJOR** – Breaking architectural or API changes.
* **MINOR** – New backward-compatible functionality.
* **PATCH** – Backward-compatible bug fixes and minor improvements.

# 7. Release Process

Before creating a release:

* All required tests pass.
* Documentation is updated.
* Security review is complete.
* Release notes are prepared.
* Version number is assigned.
* Related ADRs are finalized.

---

# 8. Version Numbering Examples

| Version | Meaning                         |
| ------- | ------------------------------- |
| v0.1.0  | Initial development milestone   |
| v0.5.0  | Major MVP progress              |
| v1.0.0  | First production release        |
| v1.1.0  | New backward-compatible feature |
| v1.1.1  | Patch release                   |
| v2.0.0  | Breaking changes                |

---

# 9. Documentation Changes

Major documentation updates should be included in the relevant release.

Examples:

* Architecture revisions.
* API contract updates.
* Security guideline updates.
* Privacy architecture revisions.
* Roadmap updates.

Minor wording corrections do not require changelog entries.

---

# 10. Release Checklist

Each release should verify:

* Architecture consistency.
* Requirements traceability.
* Test completion.
* Security validation.
* Privacy validation.
* Performance review.
* Documentation updates.
* Version increment.
* Changelog update.
